# 会場ノートPC中継方式 実施タスクリスト

## 目的
- 会場ノートPCは対局GUI/接続のみ担当する。
- 思考計算は自宅PCで実行する。
- 通信断や遅延があっても安全に対局継続できる構成にする。

## 構成（案1）
- 会場ノートPC
  - 将棋GUI
  - USIプロキシエンジン（ローカルUSI <-> リモートUSI）
- 自宅PC
  - 既存エンジン本体（`mcts_player` など）
  - リモート受信用サーバ
- 通信
  - Tailscale / WireGuard などの閉域ネットワーク推奨

## 実装タスク

### 1. 通信仕様の固定
- [ ] USI 1行をそのまま転送する方式にする（改行区切り）。
- [ ] 文字コード・改行コードを固定する（UTF-8 / LF）。
- [ ] タイムアウト値を決める（例: 読み3秒、`go` 応答は別管理）。
- [ ] ログフォーマットを決める（timestamp, direction, command, latency）。

### 2. 自宅PC側サーバ実装
- [ ] `remote_engine_server.py` を作る。
- [ ] 1接続ごとにエンジンプロセスを1つ起動する。
- [ ] stdin/stdout を中継して USI コマンドを処理する。
- [ ] `quit` 受信時にエンジンと接続を正常終了する。
- [ ] 例外時のエラー応答と再接続待ちを実装する。

### 3. 会場PC側プロキシ実装
- [ ] `usi_proxy.py` を作る。
- [ ] GUIからのUSI入力をリモートへ転送する。
- [ ] リモートからの応答をGUIへそのまま返す。
- [ ] 接続断時の挙動を定義する（即再接続、失敗時メッセージ）。
- [ ] `isready` / `go` / `stop` / `ponderhit` の動作を重点確認する。

### 4. フォールバック実装（重要）
- [ ] 接続断時にローカル代替エンジンへ切替可能にする。
- [ ] 切替条件（連続失敗回数、タイムアウト）を定義する。
- [ ] 切替後は復帰せず固定にするか、復帰を許可するか決める。
- [ ] GUIに分かる `info string` を出す。

### 5. セキュリティ/運用
- [ ] 公開インターネット直公開は避ける（閉域VPN前提）。
- [ ] 接続元を制限する（許可IP/鍵）。
- [ ] サーバ起動スクリプトを作る（再起動時の復旧含む）。
- [ ] ログ保存先とローテーションを決める。

### 6. 時間管理調整
- [ ] 往復遅延を計測し、`time_margin` / `byoyomi_margin` を上げる。
- [ ] `go` 送信から `bestmove` 受信までの実測を記録する。
- [ ] 最悪遅延時の安全側設定を準備する。

- [ ] 遅延が高いときに時間切れしないか確認。

## リスクと対策
- 回線断
  - 対策: フォールバック + 再接続 + 時間マージン拡大
- 遅延増大
  - 対策: 思考時間設定を短めにし、余裕時間を多めに確保
- プロキシ不具合
  - 対策: 最小実装に絞る、ログで再現可能にする
- 大会規約抵触
  - 対策: 事前に運営へ「外部計算資源/遠隔操作」可否を確認

## 最小実装（追加済みファイル）
- 自宅PC側: `remote/remote_engine_server.py`
- 会場PC側: `remote/usi_proxy.py`

### 起動例
- 自宅PC（サーバ）:
  - `python remote/remote_engine_server.py --host 0.0.0.0 --port 49001 --engine-command "python player/mcts_player.py" --token YOUR_TOKEN`
- 会場PC（USIプロキシ）:
  - `python remote/usi_proxy.py --host <home-pc-ip-or-tailscale-ip> --port 49001 --token YOUR_TOKEN`

### GUI設定
- エンジン実行ファイルに `python remote/usi_proxy.py ...` を登録する。
- GUIは通常USIエンジンとして扱える（stdin/stdout はプロキシが処理）。

## 本番想定：同一Wi-Fi外（会場回線と自宅回線が別）の手順

同じルータ配下（例: 自宅のLAN）にいない場合、**会場PCの `192.168.x.x` から自宅PCのローカルIPには届かない**。必ず **閉域で届くIP**（VPNの仮想IPなど）か、**意図的に穴を開けた経路**が必要。

### 推奨：閉域VPN（Tailscale / Headscale / WireGuard 等）

**方針**: 自宅PCと会場PCの双方に VPN クライアントを入れ、**仮想IP同士**で `remote_engine_server` に接続する。`usi_proxy` の `--host` には **自宅PCの VPN 上のIP** を書く（例: Tailscale なら `100.x.x.x`）。

#### 事前準備（前日まで）
- [ ] 会場PC・自宅PCの双方で同じ VPN アカウント/設定を入れる。
- [ ] 自宅PCの VPN 上IPを控える（`ping` で会場から届くかは当日前にテスト）。
- [ ] `remote_engine_server.py` を自宅PCで起動し、**Listen は `0.0.0.0`**（既存通り）でOK。接続元は **VPN 経由のみ**に相当する。
- [ ] 可能なら OS ファイアウォールで **受信を「VPN インターフェースからの 49001 のみ」**に絞る（不要な向きの開放を避ける）。
- [ ] `--token` を決め、会場の `usi_proxy` と一致させる。

#### 本番当日（会場）
- [ ] 会場PCで **VPN 接続が「接続済み」**であることを確認（切断だと疎通しない）。
- [ ] 会場PCから `ping <自宅のVPN-IP>` が通るか確認。
- [ ] 自宅PCでサーバ起動:
  - `python remote/remote_engine_server.py --host 0.0.0.0 --port 49001 --engine-command "python player/mcts_player.py" --token <token>`
- [ ] 会場PCで（テスト用）`python -c "import socket; s=socket.create_connection(('<自宅VPN-IP>',49001),5); print('ok')"` などで **TCP 49001 が届く**か確認。
- [ ] GUI のエンジンコマンドを `usi_proxy.py` にし、`--host` を **自宅の VPN IP** に固定。
- [ ] 将棋GUIで `usi` → `isready` → 短い `go` で **bestmove まで**返るか確認。

#### 遅延・時間設定
- [ ] 会場回線は往復遅延が大きいことが多い。GUI/エンジンの **持ち時間・秒読みに余裕**、エンジン側の `time_margin` / `byoyomi_margin` を大きめに。
- [ ] 可能なら前日、**会場と同種の回線（テザリング等）**で `go` 応答までの遅延を把握する。

### 非推奨だが可能：ルータのポート開放（自宅Global IP → 自宅PC:49001）

- 自宅回線の **グローバルIP + ポートフォワード**で直接届ける方法。設定は回線事業者・ルータ依存。
- **リスク**: ポートが全世界に晒される。必ず **VPN やトークンより強い対策**（接続元制限、可能ならSSHトンネル、本来はTLS）を検討。大会前にセキュリティの扱いを固める。

本番は **まず VPN で閉域** を優先し、ポート開放は最終手段とする。

### 同一Wi-Fi内との違い（一覧）

| 項目 | 同一Wi-Fi（LAN） | 別ネット（本番想定） |
|------|------------------|----------------------|
| `--host` に指定するもの | 自宅PCの LAN IP（例: `192.168.1.50`） | **VPN 仮想IP** またはポートフォワード先 |
| 追加作業 | LAN内FWでポート許可程度が多い | **VPN 接続の確立**が必須 |
| 届かない典型原因 | ゲスト隔離 / FW | **そもそもルーティングされない** |
